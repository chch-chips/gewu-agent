# 产品需求文档

## 1. 项目背景

个人学习和求职过程中会积累大量 PDF、PPT、面试题、课程资料和项目经验。通用大模型可以回答通用问题，但难以长期理解个人资料体系，也无法持续沉淀个人学习轨迹。

Gewu Agent 要解决的问题是：把个人资料、学习任务、智能体能力和工具调用整合到一个可持续演进的平台里。

## 2. 产品定位

Gewu Agent 是一个个人智能体平台，首期聚焦学习与求职场景，长期扩展为可配置、多智能体、多知识库、多工具的 AI 工作台。

项目同时服务两个目标：

- 实用目标：提升面试准备、考研复习、资料检索和知识总结效率。
- 学习目标：系统掌握 Spring AI、RAG、Tool Calling、MCP 和 ReAct Agent 的工程实现。

## 3. 目标用户

首期目标用户是项目作者本人，画像如下：

- 正在学习 Spring AI 和智能体开发。
- 需要管理面试题、408 课件、PDF/PPT 资料。
- 需要一个能长期维护、能放进简历的个人项目。
- 关注工程完整性，而不只是大模型 API 调用效果。

后续可扩展到：

- 准备面试的 Java/后端学生。
- 准备考研计算机 408 的学生。
- 需要管理个人知识库和学习助手的开发者。

## 4. 首期智能体

### 4.1 面试助手

目标：基于面试鸭等面试题资料，提供个性化面试问答、知识点解释和追问训练。

核心场景：

- 用户问某个 Java、Spring、Redis、MySQL、JVM、操作系统、计算机网络问题。
- 系统从面试知识库中检索相关片段，生成结构化回答。
- 用户请求“模拟面试”，智能体按主题连续追问。
- 用户请求“回答优化”，智能体把答案改写成面试表达。
- 用户请求“薄弱点总结”，系统根据历史会话生成复习建议。

首期验收：

- 支持 PDF 文档上传、解析、切分、向量化和检索。
- 回答中展示引用来源：文件名、页码或章节、片段摘要。
- 支持按知识库限定问答范围。
- 支持“标准答案 / 面试口述版 / 追问版”三种回答风格。

### 4.2 考研学习助手

目标：基于 408 课件和资料，提供课程知识点解释、章节总结和题目讲解。

核心场景：

- 用户询问某个知识点，例如“进程和线程的区别”。
- 系统结合课件资料生成解释，并给出关联知识点。
- 用户上传课件后，系统自动建立课程、章节、页码等元数据。
- 用户请求“按考试角度讲”，智能体给出考点、常见题型和易错点。

首期验收：

- PDF 课件可入库。
- 支持按课程维度组织知识库：数据结构、计组、操作系统、计网。
- 支持生成章节摘要。
- 支持根据问答记录生成复习清单。

### 4.3 LeManus

目标：参照 OpenManus 思路，自研一个 ReAct 智能体框架，重点学习 Agent 原理。

LeManus 不追求一开始能力最强，而追求核心循环清晰：

1. Reason: 分析任务，形成下一步计划。
2. Act: 调用工具或查询知识库。
3. Observe: 接收工具执行结果。
4. Reflect: 判断是否继续、修正还是结束。
5. Answer: 输出最终结果和执行轨迹。

首期工具：

- 知识库检索工具。
- 本地文件摘要工具。
- 时间/日期工具。
- HTTP 查询工具，默认关闭，按需授权。
- MCP 工具接入。

首期验收：

- 能完成简单多步任务，例如“根据某份资料总结知识点并生成复习计划”。
- 能展示执行轨迹，方便调试。
- 能限制最大步数、工具白名单和执行超时。

## 5. 为什么不直接把 PDF 投给在线大模型

直接上传 PDF 给在线模型适合一次性问答，但不适合构建长期平台。

Gewu Agent 的差异：

| 维度 | 直接上传 PDF 给在线模型 | 自建 Gewu Agent |
| --- | --- | --- |
| 知识复用 | 每次会话重新上传或依赖平台上下文 | 文档一次入库，长期可检索 |
| 检索控制 | 平台黑盒处理 | 自己控制切分、召回、重排、引用 |
| 个性化 | 主要依赖临时提示词 | 可沉淀学习历史、错题、偏好和智能体配置 |
| 成本 | 大文件反复输入成本高 | 入库后按检索片段输入，成本可控 |
| 可解释性 | 不一定能稳定给出出处 | 可强制展示来源片段 |
| 工程价值 | 使用工具 | 构建系统，能展示完整工程能力 |
| 隐私 | 文件交给第三方平台处理 | 可选择本地模型和本地知识库 |
| 可扩展性 | 受平台功能限制 | 可接入工具、MCP、任务流和多智能体 |

因此，项目价值不是“问答效果一定超过云端大模型”，而是“把大模型能力工程化、产品化、可控化”。

## 6. 功能边界

### 必须做

- 用户登录可后置，但系统需要预留用户维度。
- 智能体列表、智能体配置、会话管理。
- 文档上传、解析、切分、向量化、检索。
- RAG 问答，带来源引用。
- 模型配置：阶段 0 只支持 DeepSeek 云模型；本地 Ollama 后续再接。
- LeManus ReAct 核心循环。
- 工具调用和 MCP 接入的最小闭环。

### 应该做

- 文档重建索引。
- 知识库分组和标签。
- 会话历史总结。
- 回答质量反馈。
- 学习计划生成。
- PPT/PPTX 文本抽取。

### 暂不做

- 多租户商业化。
- 复杂权限系统。
- 在线支付。
- 大规模分布式训练。
- 自研向量数据库。
- 完整多模态 PPT 视觉理解。

## 7. PPT/PPTX 能力判断

可以支持读取 PPT/PPTX，但建议分阶段：

- 一期：抽取 PPT/PPTX 中的文本、备注、标题、页码，进入 RAG。
- 二期：抽取图片 OCR、表格、图示说明。
- 三期：结合多模态模型理解复杂流程图、架构图和公式。

工程建议：

- PDF：优先使用 Spring AI PDF Document Reader。
- PPT/PPTX/DOCX：使用 Apache Tika 做统一文本抽取；需要更细粒度控制时使用 Apache POI。
- 入库时统一转换成 DocumentChunk，字段包括 sourceType、fileName、pageOrSlide、section、text、metadata。

## 8. 非功能需求

- 可观测：记录模型请求、检索结果、工具调用轨迹和错误日志。
- 可评估：建立固定问答集，比较不同切分策略、模型、向量库效果。
- 可替换：模型、向量数据库、文档解析器都通过接口隔离。
- 可控成本：云模型调用应有 token 统计和限流。
- 安全：工具调用必须有白名单、超时、参数校验和执行日志。
